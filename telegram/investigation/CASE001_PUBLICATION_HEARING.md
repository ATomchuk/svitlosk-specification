# Part 2: Competing Semantic Models

---

## Model A: Publication as Message

### 1. Formal Semantic Definition

Publication IS a discrete unit of information transmitted from a sender to receivers through a communication channel.

### 2. Architectural Role

**Information Object** — not an Entity (no mutable identity), not a Value Object (has identity), not an Aggregate (no children).

Publication is a self-contained packet of information that exists to be delivered. Its essential nature is transmission.

### 3. Lifecycle

**Can Publication exist before Issue?** NO — a Message belongs to a channel/context; Issue is that context.

**Can Issue exist before Publication?** YES — a channel can exist without messages.

**Can Publication exist without Journal?** NO — Journal is the channel; without channel, no message.

**Can Publication be deleted?** YES — messages can be deleted.

**Can Publication change identity?** NO — a message is what it was when sent.

### 4. Ownership

**Owner: Telegram Channel** — the channel owns the messages it contains. The sender creates, but the channel owns.

### 5. Identity

**Two Publications are different if they are different messages** — identified by position in channel, timestamp, and content. Telegram Message ID is the natural identity.

### 6. Relationships

```text
Journal (Channel)
    │
    └──→ Issue (Temporal Context)
              │
              └──→ Publication (Message)
                        │
                        └──→ Telegram Message ID (Platform Identity)
```

### 7. Advantages

- Simple, intuitive model
- Matches common understanding of "publication"
- Explains why Telegram Message ID is part of identity
- Explains why deletions are meaningful
- Matches the forbidden terms (Message, Post)

### 8. Weaknesses

- Messages are typically transient; Publications are archived permanently
- Messages are typically not updated in place; Publications are
- Messages don't have territorial grammar; Publications do
- Messages are typically one-way; Publications have complex lifecycle
- Contradicts platform independence (TJS-000)

### 9. Repository Evidence

**Supporting:**
- GLOSSARY.md: "A public information message"
- TJS-000A §17: Forbidden terms include "Message", "Post" — suggests Publication is NOT those things
- Legacy TJS-003: Structure similar to message formatting

**Contradicting:**
- TJS-021: Complex lifecycle (Generated → Archived) — messages don't have this
- TJS-000: Platform independence — messages are platform-specific
- TJS-005: Territorial grammar — messages don't have this
- Legacy TJS-007 §6: Canonical Equality — messages don't have this

## Model B: Publication as Document

### 1. Formal Semantic Definition

Publication IS a structured, persistent, authoritative record of information that follows defined editorial standards.

### 2. Architectural Role

**Entity** — has identity (Telegram Message ID or internal ID), has mutable state (can be updated), persists over time, follows editorial rules.

Publication is a authored artifact that exists to be read and referenced.

### 3. Lifecycle

**Can Publication exist before Issue?** NO — a document belongs to a publication (journal issue).

**Can Issue exist before Publication?** YES — an issue can be planned before documents are created.

**Can Publication exist without Journal?** NO — Journal is the publication; without publication, no document.

**Can Publication be deleted?** NO — documents are preserved (archived, not deleted).

**Can Publication change identity?** NO — a document retains identity across versions.

### 4. Ownership

**Owner: Editorial System** — the editorial process creates and governs documents. The system implements editorial decisions.

### 5. Identity

**Two Publications are different if they represent different editorial units** — identified by Territory, Date, and Template. Telegram Message ID is an implementation detail, not identity.

### 6. Relationships

```text
Journal (Publication)
    │
    └──→ Issue (Edition)
              │
              └──→ Publication (Document)
                        │
                        ├──→ Territory (Scope)
                        ├──→ Template (Format)
                        └──→ Editorial Policy (Rules)
```

### 7. Advantages

- Explains why Publications are archived permanently
- Explains editorial governance (TJS-020)
- Explains territorial grammar
- Explains deterministic rendering (documents are reproducible)
- Matches DATA_MODEL.md: "Logical representation"

### 8. Weaknesses

- Documents are typically authored by humans; Publications are auto-generated
- Documents don't have Telegram Message IDs as identity
- Documents don't have lifecycle states like "Removed"
- Documents are typically static; Publications are updated in place
- Contradicts the "message" emphasis in GLOSSARY.md

### 9. Repository Evidence

**Supporting:**
- DATA_MODEL.md: "Logical representation of interpreted information"
- TJS-020: Editorial Policy governs Publications
- TJS-005: Canonical Templates define structure
- GLOSSARY.md: "Publication Lifecycle" — documents have lifecycles

**Contradicting:**
- GLOSSARY.md: "A public information message" — not "document"
- TJS-021 §4.5: "Removed" state — documents aren't removed
- Legacy TJS-007 §4: Telegram Message ID — documents don't have platform IDs
- TJS-010: Auto-generation — documents are authored

## Model C: Publication as Event

### 1. Formal Semantic Definition

Publication IS a discrete occurrence in time that represents the system communicating information to residents.

### 2. Architectural Role

**Domain Event** — marks a point in time when something happened (information became available). Has timestamp, has payload (the information), has consumers (residents).

Publication is not a thing; it is something that happens.

### 3. Lifecycle

**Can Publication exist before Issue?** NO — events occur within temporal contexts.

**Can Issue exist before Publication?** YES — temporal context exists before events occur.

**Can Publication exist without Journal?** NO — Journal is the stream of events.

**Can Publication be deleted?** NO — events cannot be un-happened (only their records can be removed).

**Can Publication change identity?** NO — an event is what it was when it occurred.

### 4. Ownership

**Owner: Publication Engine** — the engine triggers the event. Once triggered, the event exists independently.

### 5. Identity

**Two Publications are different if they occurred at different times or convey different information** — identified by timestamp and content hash. Telegram Message ID is the record of the event.

### 6. Relationships

```text
Journal (Event Stream)
    │
    └──→ Issue (Temporal Window)
              │
              └──→ Publication (Event)
                        │
                        ├──→ Trigger (Dataset Change)
                        ├──→ Timestamp (When)
                        └──→ Payload (What)
```

### 7. Advantages

- Explains event-driven architecture (TJS-008 §7: "Event-driven Publication Principle")
- Explains why time alone doesn't trigger updates
- Explains why Publications are triggered by Dataset changes
- Explains deterministic behavior (same trigger = same event)

### 8. Weaknesses

- Events are typically instantaneous; Publications persist
- Events don't have content structure; Publications do
- Events don't have territorial grammar; Publications do
- Events are typically not updated; Publications are
- Events don't have templates; Publications do

### 9. Repository Evidence

**Supporting:**
- Legacy TJS-008 §7: "Event-driven Publication Principle"
- TJS-010 §6.2: "Publication updates SHALL occur only after a new normalized Dataset becomes available"
- Legacy TJS-008 §7: "Time alone SHALL NOT trigger Publication updates"

**Contradicting:**
- TJS-021: Complex lifecycle with states — events don't have states
- TJS-005: Canonical Templates — events don't have templates
- TJS-006: Rendering Pipeline — events don't have rendering
- TJS-022: Graphic Publications — events don't have visual form

## Model D: Publication as Information Object

### 1. Formal Semantic Definition

Publication IS a self-contained unit of structured information with defined behavior, identity, and lifecycle.

### 2. Architectural Role

**Aggregate Root** — owns its lifecycle, has identity, contains internal structure, has invariants that must be preserved.

Publication is the central entity around which the entire system orbits.

### 3. Lifecycle

**Can Publication exist before Issue?** UNCLEAR — circular dependency exists in repository.

**Can Issue exist before Publication?** UNCLEAR — TJS-021 §8.1 says Issue opens when first Publication is generated.

**Can Publication exist without Journal?** YES — Publication is platform-independent; Journal is just one delivery mechanism.

**Can Publication be deleted?** CONDITIONALLY — temporary Publications can be removed; permanent Publications cannot.

**Can Publication change identity?** NO — identity is preserved across updates.

### 4. Ownership

**Owner: Publication Engine** — the engine creates and manages Publications. Ownership is determined by Telegram Message ID.

### 5. Identity

**Two Publications are different if they represent different information units** — identified by Territory + Date + Template. Telegram Message ID is the external reference, not the identity.

### 6. Relationships

```text
Publication (Aggregate Root)
    │
    ├──→ Publication Package (Collection)
    ├──→ Publication Lifecycle (State Machine)
    ├──→ Territory (Scope)
    ├──→ Canonical Template (Format)
    ├──→ Rendering Rules (Transformation)
    ├──→ Telegram Message ID (External Reference)
    └──→ Archive (Historical Record)
```

### 7. Advantages

- Explains semantic centrality (8 relationships IN, 6 OUT)
- Explains why Publication is CRITICAL in resilience analysis
- Explains why all 8 Components interact with Publication
- Explains lifecycle complexity
- Matches DDD Aggregate Root classification

### 8. Weaknesses

- Aggregate Roots typically have clear identity; Publication identity is unclear
- Aggregate Roots typically own children; Publication has no children
- Aggregate Roots are typically created by domain logic; Publications are auto-generated
- Circular dependency with Issue is problematic

### 9. Repository Evidence

**Supporting:**
- SEMANTIC_CENTER_ANALYSIS: "Publication is the semantic center"
- ONTOLOGY_RESILIENCE_REPORT: "1 CRITICAL concept (Publication)"
- DDD_AGGREGATE_ANALYSIS: "Publication is Aggregate Root"
- ONTOLOGY_REACHABILITY_REPORT: "88/88 concepts reachable"

**Contradicting:**
- No explicit identity criteria defined
- Circular dependency with Issue (TJS-000 §4, TJS-021 §8.1)
- Manual Publications excluded from ownership model
- Platform-specific (Telegram Message ID) vs platform-independent claims

## Model I: Publication as Value Object (DDD)

### 1. Formal Semantic Definition

Publication IS an immutable value defined by its attributes (Territory, Date, Template, Content) rather than by identity.

### 2. Architectural Role

**Value Object (Об'єкт-значення)** — identified by attributes, immutable, replaceable, no independent lifecycle.

### 3. Lifecycle

**Can Publication exist before Issue?** NO — Value Objects belong to Entities.

**Can Issue exist before Publication?** YES — Entity exists before its Value Objects.

**Can Publication exist without Journal?** NO — Journal is the containing Entity.

**Can Publication be deleted?** YES — Value Objects can be replaced.

**Can Publication change identity?** NO — Value Objects have no identity to change.

### 4. Ownership

**Owner: Issue** — the Issue Entity owns its Publications as Value Objects.

### 5. Identity

**Two Publications are different if they have different attributes** — identified by Territory + Date + Template + Content hash. No separate identity.

### 6. Relationships

```text
Journal (Aggregate Root)
    │
    └──→ Issue (Entity)
              │
              └──→ Publication (Value Object)
                        │
                        ├──→ Territory (Attribute)
                        ├──→ Date (Attribute)
                        ├──→ Template (Attribute)
                        └──→ Content (Attribute)
```

### 7. Advantages

- Explains why Publications are deterministic (same attributes = same Value)
- Explains Canonical Equality (byte-for-byte comparison)
- Explains why two identical Datasets produce identical Publications
- Simple identity model (attributes = identity)

### 8. Weaknesses

- Value Objects are typically immutable; Publications are updated in place
- Value Objects don't have lifecycle states; Publications do
- Value Objects don't have Telegram Message IDs; Publications do
- Value Objects are typically replaced, not updated; Publications are updated
- Manual Publications contradict Value Object semantics

### 9. Repository Evidence

**Supporting:**
