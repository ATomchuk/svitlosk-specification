## What Is Publication?

**From GLOSSARY.md:** "A public information message generated by the SvitloSk system for distribution through one or more supported communication channels."

**From DATA_MODEL.md:** "Logical representation of interpreted information prepared for public distribution."

**From TJS-000 §4:** "One independent publication belonging to an Issue."

---

## What Does Publication Own?

**From TJS-000 §5:** "Publication — Owns: Content, structure, lifecycle"

**From TJS-021 §4:** Publication owns its lifecycle states (Generated, Published, Updated, Archived, Removed)

---

## Who Creates Publication?

**From TJS-021 §6.1:** "Publications are created automatically from normalized data"

**From TJS-010 §4.1:** "Publication Engine — Transforms normalized dataset into deterministic Publication Package"

**Creator:** Publication Engine

---

## Who Destroys Publication?

**From TJS-021 §5.5:** "A Published Publication becomes Removed only if it is a temporary Publication"

**From TJS-021 §7.3:** "Deletion SHALL occur only for temporary Publications"

**Destroyer:** Publication Engine (for temporary Publications only)

---

## Can Anything Exist Without Publication?

**From TJS-000 §4:** Journal exists independently. Issue exists (but requires Publication to open).

**From CASE-002A:** Issue cannot exist without Publication.

**From TJS-010 §4.1:** Publication Engine exists without producing Publications (but has no purpose).

**Answer:** Journal can exist. Issue cannot. Publication Engine can exist but has no purpose.

---

## Can the System Exist Without Publication?

**From TJS-010 §3.3:** Data flow: Parser → Publication Engine → Publication Package → Telegram Publisher → Telegram Channel → Subscribers

**Analysis:** If Publication is removed, the entire data flow breaks. Parser produces data, but nothing receives it. Telegram Publisher has nothing to deliver. Subscribers receive nothing.

**Answer:** NO — the system cannot function without Publication.

# Part II: Three Worlds

## WORLD A: Publication Exists (Current State)

**Architecture:**

```text
Parser → Normalized Dataset → Publication Engine → Publication Package → Telegram Publisher → Telegram Channel → Subscribers
```

**Lifecycle:**

```text
Generated → Published → Updated → Archived → Removed
```

**Identity:** Territory + Date + Template

**Ownership:** Publication Engine

**Status:** WORKING

---

## WORLD B: Publication Does NOT Exist — Replace with Issue

**Proposed Architecture:**

```text
Parser → Normalized Dataset → Publication Engine → Issue Package → Telegram Publisher → Telegram Channel → Subscribers
```

**Changes Required:**

1. Rename "Publication" to "Issue" throughout
2. Merge Publication and Issue into single concept
3. Remove Publication Engine (Issue becomes the primary concept)
4. Update all lifecycle states to apply to Issue
5. Update all ownership to apply to Issue
6. Update all identity to apply to Issue

**Consequences:**

| Aspect | Change |
|--------|--------|
| Semantic hierarchy | Journal → Issue (no Publication level) |
| Lifecycle | Issue has Generated, Published, Updated, Archived, Removed |
| Identity | Issue identified by Calendar day |
| Ownership | Issue owned by Publication Engine |
| Telegram delivery | Issue delivered directly to Telegram |

**Status:** REQUIRES MAJOR RESTRUCTURING

---

## WORLD C: Publication Becomes Only Representation

**Proposed Architecture:**

```text
Parser → Normalized Dataset → Publication Engine → [Telegram Message | Repository Entry | Archive Entry]
```

**Changes Required:**

1. Remove "Publication" as domain concept
2. Telegram Message, Repository Entry, Archive Entry become the primary concepts
3. Each has its own lifecycle
4. Each has its own identity
5. Each has its own ownership

**Consequences:**

| Aspect | Change |
|--------|--------|
| Domain concepts | No Publication — only platform artifacts |
| Lifecycle | Each representation has its own lifecycle |
| Identity | Each representation has its own identity |
| Ownership | Each representation owned by different component |
| Consistency | No single concept across representations |

**Status:** FRAGMENTS THE DOMAIN

# Part V: Removal Experiments

## Attempt to Remove Publication

**Result:** IMPOSSIBLE

**Why:**

1. Publication is the output of the entire data flow
2. Without Publication, Parser has no consumer
3. Without Publication, Telegram Publisher has nothing to deliver
4. Without Publication, Subscribers receive nothing
5. Without Publication, the system has no purpose

**Evidence:** TJS-010 §3.3 defines the complete data flow ending at Subscribers. Publication is the central object in this flow.

---

## Attempt to Replace Publication

**Result:** DIFFICULT

**Why:**

1. Publication has unique identity (Territory + Date + Template)
2. Publication has unique lifecycle (Generated → Published → Updated → Archived → Removed)
3. Publication has unique ownership (Publication Engine)
4. Publication is the semantic center of the ontology
5. No other concept can replace all these properties

**Evidence:** CASE-001 investigations confirm Publication as semantic center.

---

## Attempt to Merge Publication into Issue

**Result:** PROBLEMATIC

**Why:**

1. Publication and Issue have different identities (Territory + Date + Template vs Calendar day)
2. Publication and Issue have different lifecycles
3. Publication and Issue have different ownership
4. Merging would lose the territorial granularity of Publication
5. Issue is DERIVATIVE — it depends on Publication

**Evidence:** CASE-002A investigations confirm Issue depends on Publication.

---

## Attempt to Replace Publication with Telegram Message

**Result:** IMPOSSIBLE

**Why:**

1. Telegram Message is a REPRESENTATION, not a domain concept
2. Telegram Message has platform-specific identity (Telegram Message ID)
3. Telegram Message cannot exist before delivery
4. Telegram Message is temporary (can be deleted)
5. Publication is platform-independent; Telegram Message is platform-dependent

**Evidence:** TJS-000 §3: "Telegram itself SHALL NOT be considered the Journal"

# Part VII: Key Findings

## Finding 1: Publication Is Fundamental

**Evidence:**

- Publication is the output of the entire data flow
- Publication is the semantic center of the ontology
- Publication has unique identity, lifecycle, ownership
- System cannot function without Publication

---

## Finding 2: Publication Is Not Artificial

**Evidence:**

- Publication has real-world meaning (public information message)
- Publication exists independently of implementation
- Publication is defined in multiple normative documents
- Publication is consistent across all specifications

---

## Finding 3: Publication Cannot Be Removed

**Evidence:**

- Removing Publication breaks the data flow
- Removing Publication breaks the lifecycle
- Removing Publication breaks the identity model
- Removing Publication breaks the ownership model

---

## Finding 4: Publication Cannot Be Replaced

**Evidence:**

- No other concept has all Publication properties
- Issue is derivative (depends on Publication)
- Telegram Message is a representation (not domain concept)
- Publication Package is a collection (not individual concept)
