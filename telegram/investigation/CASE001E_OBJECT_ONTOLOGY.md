# Part 3: Object Taxonomy

## Categories Found in Repository

### Category 1: Reference Entities

**Definition:** Stable entities describing the operational environment.

**Evidence:** DATA_MODEL.md § "Reference Entities"

**Examples:** Address, Queue, Subqueue

**Characteristics:**

- Static (do not change)
- Predate SvitloSk
- Official designation
- Traceable to real world

---

### Category 2: Operational Entities

**Definition:** Entities representing official operational information.

**Evidence:** DATA_MODEL.md § "Operational Entities"

**Examples:** Planned Power Outage, Emergency Power Outage, Schedule

**Characteristics:**

- Dynamic (change over time)
- Originate from DSO
- Official information
- Time-bound

---

### Category 3: Publication Entities

**Definition:** Entities created from interpreted operational information.

**Evidence:** DATA_MODEL.md § "Publication Entities"

**Examples:** Publication, Publication Package

**Characteristics:**

- Created by system
- Derived from operational data
- For public distribution
- Deterministic

---

### Category 4: Archive Entities

**Definition:** Entities supporting historical preservation.

**Evidence:** DATA_MODEL.md § "Archive Entities"

**Examples:** Daily Snapshot, Analytics Record

**Characteristics:**

- Permanent
- Immutable
- Historical
- Read-only

---

### Category 5: Architectural Components

**Definition:** Concrete architectural building blocks.

**Evidence:** GLOSSARY.md §7: "A Component is a concrete architectural building block"

**Examples:** Publication Engine, Parser, Telegram Publisher

**Characteristics:**

- Implementation units
- Have stable architectural identity
- Participate in system architecture
- NOT business concepts

---

### Category 6: Logical Roles

**Definition:** Logical responsibilities performed within the system.

**Evidence:** GLOSSARY.md §7: "A Role is a logical responsibility"

**Examples:** Publisher, Interpreter

**Characteristics:**

- Implementation-independent
- Describe behavior
- May be implemented by different Components
- NOT architectural Components

---

### Category 7: Normative Documents

**Definition:** Documents defining mandatory project requirements.

**Evidence:** GLOSSARY.md §1: "A document defining mandatory project requirements"

**Examples:** GLOSSARY.md, DATA_MODEL.md, TJS-021

**Characteristics:**

- Define requirements
- Stable
- Traceable
- Governed

---

### Category 8: Platform Artifacts

**Definition:** Artifacts specific to Telegram platform.

**Evidence:** TJS-000A §10

**Examples:** Telegram Message, Telegram Channel

**Characteristics:**

- Platform-specific
- Technical identifiers
- Transient (can be deleted)
- Borrow identity from domain objects

# Part 6: Object vs Identity

## Investigation

**Does Object own Identity?**

**Evidence:**

- Publication has Identity (Territory + Date + Template) that persists across lifecycle
- Territory has Identity (Administrative name) that exists independently
- Telegram Message borrows Identity from Publication

**Conclusion:** Object owns Identity (for most objects). Some objects borrow Identity.

---

**Does Identity own Object?**

**Evidence:**

- Identity (Territory + Date + Template) defines what Publication IS
- But Identity exists without Object (concept exists before implementation)

**Conclusion:** Neither owns the other. They are complementary aspects.

---

**Can Object exist without Identity?**

**Evidence:**

- Telegram Message has no intrinsic Identity (borrows from Publication)
- Publication Package has no intrinsic Identity (borrows from Reporting Period)

**Conclusion:** YES — some objects exist without intrinsic Identity.

---

**Can Identity exist without Object?**

**Evidence:**

- Territory Identity (Administrative name) exists in reality before any system
- Address Identity (Official designation) exists before any system

**Conclusion:** YES — Identity can exist without Object (for real-world objects).

# Part 9: Minimal Ontology

## Smallest Possible Ontology

**Attempt 1: Remove all categories except "Object"**

**Result:** FAILED — cannot distinguish between Publication (business artifact) and Telegram Message (platform representation).

**Attempt 2: Merge Reference Entities and Operational Entities**

**Result:** FAILED — Reference Entities are static, Operational Entities are dynamic.

**Attempt 3: Merge Publication Entities and Archive Entities**

**Result:** FAILED — Publication Entities are active, Archive Entities are permanent.

**Attempt 4: Merge Architectural Components and Logical Roles**

**Result:** FAILED — Components are implementation units, Roles are responsibilities.

**Attempt 5: Remove Platform Artifacts**

**Result:** FAILED — Telegram Message is essential for delivery.

**Minimal Ontology:**

1. **Real-World Objects** — exist independently of SvitloSk
2. **Domain Objects** — created by SvitloSk business logic
3. **Information Objects** — created by SvitloSk data processing
4. **Representation Objects** — projections of domain objects
5. **Architectural Objects** — system building blocks

# Part 10: Contradictions

## Contradiction 1: Entity vs Object

**Evidence:**

- DATA_MODEL.md uses "Entity" (Reference Entities, Operational Entities, Publication Entities)
- Previous investigations use "Object" (Publication, Territory, Address)

**Severity:** LOW

**Possible Explanation:** "Entity" is the data model term, "Object" is the ontological term.

---

## Contradiction 2: Representation Treated as Object

**Evidence:**

- Telegram Message is sometimes called an "object" but is actually a representation
- Historical Record is sometimes called an "object" but is actually a representation

**Severity:** MEDIUM

**Possible Explanation:** Representations are often colloquially called "objects" but are ontologically distinct.

---

## Contradiction 3: Component vs Object

**Evidence:**

- GLOSSARY.md §7: "A Component SHALL NOT be considered a business concept"
- But Components are often discussed alongside domain objects

**Severity:** LOW

**Possible Explanation:** Components are architectural objects, not business objects.

---

## Contradiction 4: Identity Confusion

**Evidence:**

- Some objects have intrinsic Identity (Publication, Territory)
- Some objects borrow Identity (Telegram Message, Historical Record)
- Some objects have no Identity (Publication Package)

**Severity:** MEDIUM

**Possible Explanation:** Identity is not a universal property of all objects.

---

## Contradiction 5: Lifecycle Confusion

**Evidence:**

- Some objects have complex lifecycles (Publication)
- Some objects have simple lifecycles (Territory)
- Some objects have no lifecycle (Address)

**Severity:** LOW

**Possible Explanation:** Lifecycle complexity varies by object type.

---

## Contradiction 6: Ownership Confusion

**Evidence:**

- Publication Engine owns Publications
- Parser owns Normalized Dataset
- But who owns Territory? Address? Queue?

**Severity:** MEDIUM

**Possible Explanation:** Real-world objects have no owner in the system.
