# Purpose

# Identity Ownership Analysis

## Candidate 1: Parser

**Owns Identity?** NO

**Evidence:**

- GLOSSARY.md §2: "Parser — A software Component responsible for retrieving Open Data from approved Data Sources"
- TJS-010 §4.2: "Parser — Retrieves Open Data from approved Data Sources"
- Parser produces Normalized Dataset, not identity

**Conclusion:** Parser produces INPUT for identity creation but does not own identity.

---

## Candidate 2: Publication Engine

**Owns Identity?** **YES**

**Evidence:**

- TJS-010 §4.1: "Publication Engine — Transforms normalized dataset into deterministic Publication Package. Owns: Publication generation, distribution, ownership model"
- TJS-021 §6.1: "Publications are created automatically from normalized data. Every Publication SHALL represent exactly one Territory, use one Canonical Template"
- Publication Engine creates Publication with identity (Territory + Date + Template)

**Conclusion:** Publication Engine owns identity. It creates Publication with identity.

---

## Candidate 3: Publisher Role

**Owns Identity?** NO

**Evidence:**

- GLOSSARY.md §3: "Publisher — The logical Role responsible for preparing and distributing Publications"
- ADR-001: "Component vs Role — A Component and a Role SHALL NOT be considered interchangeable"
- Publisher is a logical role, not an identity owner

**Conclusion:** Publisher is a logical role that describes behavior, not identity ownership.

---

## Candidate 4: Telegram Publisher

**Owns Identity?** NO

**Evidence:**

- TJS-010 §4.4: "Telegram Publisher — Telegram-specific implementation of Publisher Role"
- Telegram Publisher delivers Publications, does not create identity
- Telegram Publisher assigns Telegram Message ID (external reference), not intrinsic identity

**Conclusion:** Telegram Publisher is a delivery mechanism that assigns external references, not identity.

---

## Candidate 5: Data Storage

**Owns Identity?** NO

**Evidence:**

- TJS-010 §4.7: "Data Storage — Preserving normalized data, publications, schedules, and historical records"
- Data Storage persists identity, does not create it
- Data Storage references Publications by ID, does not own identity

**Conclusion:** Data Storage is a persistence mechanism that stores identity, does not own it.

---

## Candidate 6: Issue

**Owns Identity?** NO

**Evidence:**

- TJS-000 §4: "Issue represents one editorial edition of the Journal for a single calendar day"
- TJS-021 §8: "An Issue opens when the first Publication for a calendar day is generated"
- Issue is a container, not an identity owner

**Conclusion:** Issue is a temporal container that contains Publications, does not own identity.

---

## Candidate 7: Journal

**Owns Identity?** NO

**Evidence:**

- TJS-000 §4: "Journal represents the continuous informational publication"
- GLOSSARY.md §3: "The Telegram Journal publishes Publications generated by the Publisher"
- Journal is the ultimate container, not an identity owner

**Conclusion:** Journal is the ultimate container that contains Issues, does not own identity.

---

# Identity Ownership Summary

| Component | Owns Identity? | Evidence |
|-----------|---------------|----------|
| Parser | NO | Produces input, not identity |
| Publication Engine | **YES** | Creates Publication with identity |
| Publisher Role | NO | Logical role, not identity owner |
| Telegram Publisher | NO | Delivery mechanism, assigns external references |
| Data Storage | NO | Persistence mechanism, stores identity |
| Issue | NO | Container, contains Publications |
| Journal | NO | Ultimate container, contains Issues |

---

# Key Insight

**Publication Engine is the ONLY Component that owns identity.**

**No other Component can create, modify, or destroy identity.**

**All other Components either:**

- Produce input for identity creation (Parser)
- Assign external references (Telegram Publisher)
- Store identity (Data Storage)
- Contain identity (Issue, Journal)
