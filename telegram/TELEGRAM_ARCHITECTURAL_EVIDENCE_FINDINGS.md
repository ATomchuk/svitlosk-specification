## Special Audit: Publication Lifecycle (TJS-002, TJS-007, TJS-008)

### Question: Is "Publication Lifecycle" ONE responsibility or SEVERAL independent responsibilities?

### Evidence

**TJS-002 §1 Purpose:**
> "This specification defines the complete lifecycle of publications within the official SvitloSk Telegram Journal."

**TJS-007 §1 Purpose:**
> "This specification defines the complete lifecycle of Publications within the SvitloSk Telegram Journal."

**TJS-008 §1 Purpose:**
> "This specification defines the lifecycle of Telegram Journal Publications generated by the SvitloSk Publication Engine."

### Analysis

All three documents claim to define "the lifecycle" of Publications. The word "complete" appears in TJS-002 and TJS-007 but not TJS-008. The scope is slightly different:
- TJS-002: "publications within the official SvitloSk Telegram Journal"
- TJS-007: "Publications within the SvitloSk Telegram Journal"
- TJS-008: "Telegram Journal Publications generated by the SvitloSk Publication Engine"

### What each document uniquely defines

**TJS-002 UNIQUE content (not in TJS-007 or TJS-008):**
- Publication type taxonomy (§4): 7 categories (Daily schedule, Schedule update, Emergency outage notification, Emergency restoration notification, Informational notice, Technical notice, System status message)
- Daily schedule lifecycle flow (§5): Generated → Validated → Published → Updated → Archived
- Emergency publication lifecycle flow (§6): Generated → Published immediately → Updated → Archived
- Archive policy (§9): "Archive entries shall not be modified except to correct verified technical errors"
- Deletion policy (§10): Prohibited for daily schedules, emergency notifications, restoration notifications, official system notices
- Traceability (§11): source dataset, publication timestamp, generator version, generated graphic
- Reliability (§12): no duplicates, correct order, correct targeting, archived accessibility

**TJS-007 UNIQUE content (not in TJS-002 or TJS-008):**
- Publication creation rules (§4): one territory, one template, one rendering, Telegram Message ID
- Canonical equality check (§6): compare before updating
- Publication ordering (§7): deterministic, Centre → Districts alphabetical
- Publication ownership (§10): Telegram Message ID based, only owned may be updated/removed
- Non-destructive channel principle (§11): Publisher only touches its own posts
- Lifecycle diagram (§12): operational flow
- Error handling (§13): never corrupt, never delete permanent, never modify admin posts, recoverable

**TJS-008 UNIQUE content (not in TJS-002 or TJS-007):**
- Daily publication cycle (§3): 6-step cycle
- Daily publication package (§4): Today's Package
- Tomorrow publication package (§5)
- Publication windows (§6): 05:00, 12:00, throughout day, next day 05:00
- Event-driven publication principle (§7): data triggers, not time
- Stable journal principle (§8): visual stability, update in place
- Deterministic daily journal principle (§9): identical input → identical output
- Non-destructive update principle (§10): update in place, don't recreate
- Manual publications (§13): outside lifecycle
- Image publications (§14): outside lifecycle
- Publication session (§15): one execution = one state
- Daily lifecycle diagram (§16): temporal cycle
- Architecture diagram (§17): component architecture

### Verdict

**Publication Lifecycle is ONE responsibility that has been incorrectly split into THREE documents.**

The evidence:
1. All three documents use the SAME title: "Publication Lifecycle"
2. All three Purpose statements claim to define "the lifecycle"
3. The overlapping content (lifecycle states, update rules, temporary publications) is the SAME responsibility defined with DIFFERENT details
4. The unique content in each document does NOT justify separate documents — it justifies SECTIONS within a single document

**If independent:** The responsibilities would need to have DIFFERENT purposes, DIFFERENT scopes, and NO overlapping content. This is NOT the case — all three define lifecycle states, update rules, and temporary/permanent classification.

**If identical:** The document that SHALL own it is the one with the MOST COMPLETE and MOST AUTHORITATIVE content. Based on the evidence:
- TJS-008 has the MOST unique content (daily cycle, windows, principles, session, diagrams)
- TJS-007 has IMPORTANT operational content (ownership, non-destructive channel, error handling)
- TJS-002 has IMPORTANT conceptual content (publication types, traceability, reliability)

**The merged document should be TJS-005 (Publication Lifecycle)** as proposed, containing ALL content from all three documents.
