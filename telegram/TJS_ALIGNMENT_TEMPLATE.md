# Purpose

This document defines the canonical structure that every aligned Telegram specification SHALL follow.

This template is the reference model for all TJS documents after Alignment. No aligned specification shall invent its own structure.

---

# Alignment Rules

The following rules govern every aligned Telegram specification:

1. A specification SHALL NOT define terminology. Terminology is defined exclusively by TELEGRAM_GLOSSARY.md.

2. A specification SHALL reference the Glossary when using canonical terms.

3. A specification SHALL own exactly one responsibility.

4. A specification SHALL NOT duplicate another specification's responsibility.

5. A specification SHALL NOT define business rules owned by another specification.

6. Implementation SHALL NOT appear inside conceptual specifications.

7. Specifications SHALL use RFC 2119 terminology (SHALL, SHOULD, MAY) for normative requirements.

8. Specifications SHALL declare their dependencies explicitly.

9. Specifications SHALL declare who references them.

10. Specifications SHALL maintain consistent metadata format.

11. Specifications SHALL not introduce semantic concepts not present in the Glossary.

12. Specifications SHALL reference the Glossary for every canonical term used.

# Section Definitions

## Purpose

**Required:** YES

**Content:** A concise statement of what this specification defines. Maximum 2-4 sentences.

**Style:** Use RFC 2119 terminology. Describe WHAT, not HOW.

**Example:**

```markdown
# Purpose

This specification defines the canonical publication templates used by the
SvitloSk Telegram Journal.

It establishes the normative structure, hierarchy, formatting and validation
rules for every Telegram Publication generated by the Publication Engine.
```

---

## Scope

**Required:** YES

**Content:** What this specification covers and what it intentionally excludes.

**Style:** Use bullet lists. Be explicit about exclusions.

**Example:**

```markdown
# Scope

This specification applies to:

- City Today publications;
- District Today publications;
- City Tomorrow publications;
- District Tomorrow publications.

This specification does NOT apply to:

- Rendering rules (TJS-004);
- Publication lifecycle (TJS-005);
- Channel management (TJS-006).
```

## Concept Coverage

**Required:** YES

**Content:** The mandatory Semantic Concepts required by this specification. This section SHALL reference TELEGRAM_GLOSSARY.md. Only references — no definitions. Definitions belong only to TELEGRAM_GLOSSARY.md.

**Style:** Bullet list with term and brief context. Each term SHALL be traceable to the Glossary.

**Example:**

```markdown
# Concept Coverage

This specification requires the following Semantic Concepts from TELEGRAM_GLOSSARY.md:

- **Journal** — the continuous informational publication that this specification serves
- **Issue** — the daily edition containing Publications
- **Publication** — the discrete unit of information delivered to readers
- **Publication Grammar** — the structural rules governing Publication organization
- **Canonical Templates** — the definitive publication templates (A, B, C, D)
- **Territory Header** — the beginning of each Publication
- **Publication Sections** — Planned Outages and Emergency Outages
- **Validation Rules** — the prohibitions ensuring structural conformance
```

## Semantic Boundaries

**Required:** YES

**Content:** This section SHALL explicitly state what the specification owns, uses, references, must NOT define, and must NOT duplicate. This section SHALL become the primary Separation of Concerns declaration.

**Style:** Structured list with clear categories.

**Example:**

```markdown
# Semantic Boundaries

## Owns

- Canonical publication templates (A, B, C, D)
- Structural grammar rules
- Territory header formatting
- Publication section ordering
- Validation prohibitions

## Uses

- **Publication** (from Glossary)
- **Publication Grammar** (from Glossary)
- **Territory Header** (from Glossary)
- **Validation Rules** (from Glossary)

## References

- TELEGRAM_GLOSSARY.md — canonical terminology
- TELEGRAM_SEMANTIC_MODEL.md — semantic foundation
- TJS-002 — Editorial Policy (formatting standards)

## Must NOT Define

- Editorial policy (owned by TJS-002)
- Rendering rules (owned by TJS-004)
- Publication lifecycle (owned by TJS-005)
- Any semantic concepts (owned by Glossary)

## Must NOT Duplicate

- Publication structure from TJS-003 (this specification owns it)
- Formatting rules from TJS-002 (this specification implements them)
- Rendering pipeline from TJS-004 (this specification does not define it)
```

# Normative Language Rules

## RFC 2119 Usage

| Term | Meaning | Use When |
|------|---------|----------|
| SHALL | Mandatory requirement | Defining mandatory behaviour |
| SHOULD | Recommended action | Recommending but not requiring |
| MAY | Optional action | Allowing but not requiring |

## Writing Style

- Use active voice where possible
- Use present tense for definitions
- Use RFC 2119 for normative requirements
- Keep sentences short (maximum 25 words)
- Use bullet lists for multiple items
- Use tables for structured data
