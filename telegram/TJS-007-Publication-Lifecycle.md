# Purpose

This specification defines the complete lifecycle of Telegram Publications generated by SvitloSk.

It establishes the normative rules governing:

- Publication Package generation;
- publication creation;
- publication updates;
- publication removal;
- package synchronization;
- transaction execution;
- deterministic behaviour.

This document applies exclusively to the Telegram Journal.

---

# Scope

This specification governs:

- Publication Packages;
- publication synchronization;
- publication transactions;
- publication updates;
- publication deletion;
- Telegram state management.

This specification does not define:

- data collection;
- data normalization;
- message rendering;
- editorial formatting.

---

# Fundamental Principle

The Telegram Journal SHALL always represent the current official state of the normalized Dataset.

Telegram SHALL NEVER become the system source of truth.

---

# Publication Package

A Publication Package is the complete collection of Telegram Publications generated from one normalized Dataset.

A Publication Package SHALL:

- represent exactly one Dataset;
- contain every required Publication;
- contain no obsolete Publications;
- be internally consistent.

One Dataset SHALL generate exactly one Publication Package.

---

# Canonical Package

Publication Package generation SHALL be deterministic.

Identical Datasets SHALL always produce identical Publication Packages.

Differences caused by execution time, Telegram identifiers or processing order SHALL NOT affect the generated Package.

---

# Publication Lifecycle

Every Publication Package SHALL pass through the following lifecycle:

```text
Dataset

↓

Package Generation

↓

Package Comparison

↓

Transaction Planning

↓

Telegram Synchronization

↓

Completed State
```

---

# Package Comparison

Publisher SHALL compare:

- Previous Publication Package;
- Current Publication Package.

Publisher SHALL NOT compare Telegram messages directly.

Telegram is considered an output target only.

---

# Publication Transaction

A Publication Transaction is the atomic synchronization process between the current Publication Package and Telegram.

A Transaction consists of three ordered phases:

```text
Create

↓

Edit

↓

Delete
```

The execution order SHALL NEVER change.

---

# Create Rules

A Publication SHALL be created when:

- it exists in the current Publication Package;
- it does not exist in Telegram.

---

# Update Rules

A Publication SHALL be updated only when:

- its Canonical Representation changes.

Publisher SHALL NOT update Publications because of:

- generation timestamp;
- processing time;
- Telegram identifiers;
- execution order.

---

# Delete Rules

A Publication SHALL be removed when it no longer belongs to the current Publication Package.

Typical examples include:

- Tomorrow Publications after expiration;
- obsolete emergency Publications;
- Publications removed because official data changed.

---

# State Synchronization

Telegram SHALL be treated as a synchronized projection of the current Publication Package.

Publisher SHALL continuously synchronize Telegram until both states are identical.

---

# Telegram State

Telegram SHALL NEVER be treated as project storage.

The official source of truth SHALL remain the normalized Dataset.

The following architecture is normative:

```text
Parser

↓

Dataset

↓

Publication Package

↓

Telegram Journal
```

The reverse flow SHALL NEVER occur.

---

# Deterministic Operations

Publisher SHALL execute synchronization in deterministic order.

Operations SHALL always follow:

1. Create
2. Edit
3. Delete

The execution order SHALL remain stable across all executions.

---

# Idempotency

Running Publisher multiple times using the same normalized Dataset SHALL produce an identical Telegram state.

Repeated execution SHALL NOT introduce additional edits.

---

# Canonical Equality

Two Publications SHALL be considered equal when their Canonical Representation is identical.

The following properties SHALL NOT affect equality:

- Telegram message identifier;
- publication timestamp;
- processing timestamp;
- synchronization time.

Only rendered publication content SHALL determine equality.

---

# Change Detection

Publisher SHALL modify Telegram Publications only when:

- the normalized Dataset changes; or
- the Rendering Rules change.

No other condition SHALL trigger publication updates.

---

# Publication Integrity

Every Publication Package SHALL represent one internally consistent Dataset.

Publications generated from different Dataset versions SHALL NEVER coexist inside the same Publication Package.

---

# Failure Recovery

If synchronization fails, Publisher SHALL retry synchronization.

Publisher SHALL NOT publish incomplete Publication Packages.

Publisher SHALL preserve Telegram consistency.

---

# No Manual Recovery

Manual editing of Telegram messages SHALL NOT become part of the synchronization process.

Publisher SHALL always restore Telegram to the canonical Publication Package.

---

# Future Compatibility

The Publication Lifecycle defined by this specification SHALL remain channel-independent.

Future publication channels MAY reuse the same lifecycle.

Examples include:

- PWA;
- Web Portal;
- Push Notifications;
- Email;
- Public API.

---

# Architectural Principle

Telegram is a projection of the current Publication Package, not the storage of project state.

All project state SHALL originate from the normalized Dataset.

---

# Repository Rule

All Publication lifecycle implementations SHALL conform to this specification.

Implementations SHALL NOT introduce additional synchronization behaviour outside this document.
