# 3. Analysis

## 3.1 Candidate Objects

| Candidate | Evidence Support | Weakness |
|-----------|-----------------|----------|
| Journal | CHARTER, GLOSSARY, EDITORIAL | Journal is a container, not the atomic product |
| Publication | GLOSSARY, DATA_MODEL | Publication is one message, not the complete daily product |
| Issue | LIFECYCLE | Issue is defined only in Telegram context |
| Release | User context | Not defined in any canonical document |
| Edition | EDITORIAL | Used but not formally defined |
| Information Package | None | Not defined anywhere |

## 3.2 Structural Analysis

The project produces a DAILY PRODUCT that contains MULTIPLE PUBLICATIONS organized by TERRITORY.

Evidence from user context:
```
One Journal Release contains:
- Emergency outage publication (today)
- Planned outage publication (today)
- Planned outage publication (tomorrow) — temporary
- Graph of planned power schedule for tomorrow (12 subqueues)
- Technical publication (last update time)
- Future service publications
```

This is NOT a single Publication.

This is a COLLECTION of Publications for one Reporting Period.

## 3.3 The Container Problem

The Glossary defines "Publication Package" as "The complete collection of Publications generated during one Reporting Period."

But "Publication Package" is a TECHNICAL term, not the domain product.

The domain product is the JOURNAL itself — the daily edition that residents read.

# 4. Primary Object Identification

## 4.1 The Primary Object

The primary object produced by the SvitloSk Project is:

**The Journal Release**

A Journal Release is one complete daily edition of the SvitloSk Public Information Journal.

## 4.2 Definition

> **Journal Release:** A complete daily edition of the SvitloSk Public Information Journal, containing all Publications for one Reporting Period, organized by Territory, ready for distribution through supported Communication Channels.

## 4.3 Evidence Basis

| Evidence Source | Support |
|-----------------|---------|
| CHARTER.md | "Public Information Journal" is the primary communication channel |
| GLOSSARY.md | "Publication Package" is the collection for one Reporting Period |
| User context | "Journal Release" is the product containing multiple publication types |
| EDITORIAL | "Issue" represents one editorial edition for one day |

## 4.4 Why Not "Publication"?

A Publication is ONE message. The project does NOT produce one message per day. It produces a COLLECTION of messages organized by Territory. The Publication is a PART of the Journal Release, not the Journal Release itself.

## 4.5 Why Not "Journal"?

The Journal is the ONGOING entity — the continuous publication stream. A Journal Release is one INSTANTIATION of the Journal for one day. The Journal is the abstract concept; the Journal Release is the concrete product.

## 4.6 Why Not "Issue"?

"Issue" is used in the Telegram lifecycle specification but is Telegram-specific. The concept of "Issue" needs to be generalized to "Journal Release" to be channel-independent.

# 7. Cross-Examination

## 7.1 Model A: Publication as Primary Object

**Claim:** The primary object is the Publication.

**Attack:**
- A Publication is ONE message for ONE Territory
- The project produces MULTIPLE Publications per day
- The Publications are organized into a COLLECTION
- The COLLECTION is what residents consume
- Therefore Publication is a PART, not the PRIMARY object

**Verdict:** REJECTED. Publication is an atomic unit, not the primary product.

## 7.2 Model B: Publication Package as Primary Object

**Claim:** The primary object is the Publication Package (from Glossary).

**Attack:**
- "Publication Package" is a technical term defined in the Glossary
- It describes the COLLECTION but not the DOMAIN meaning
- Residents do not think in terms of "packages"
- Residents think in terms of "today's information"
- The domain concept is "Journal Release", not "Publication Package"

**Verdict:** PARTIALLY ACCEPTED. Publication Package is the technical container; Journal Release is the domain object.

## 7.3 Model C: Journal Release as Primary Object

**Claim:** The primary object is the Journal Release.

**Attack:**
- "Journal Release" is not defined in any canonical document
- It is a NEW term introduced by this investigation

**Defense:**
- The concept EXISTS in the project (CHARTER, GLOSSARY, EDITORIAL)
- The term "Issue" is used in Telegram lifecycle but is Telegram-specific
- "Journal Release" generalizes "Issue" to be channel-independent
- The concept is PROVEN by evidence from multiple sources

**Verdict:** ACCEPTED as the primary object.

# 8. Findings

## 8.1 Finding OBJ-001

**The primary object produced by the SvitloSk Project is the Journal Release.**

A Journal Release is one complete daily edition of the SvitloSk Public Information Journal.

**Evidence:** CHARTER.md, GLOSSARY.md, TELEGRAM_EDITORIAL_SYSTEM_CANONICAL_MODEL.md, User context.

**Confidence:** HIGH.

## 8.2 Finding OBJ-002

**A Journal Release contains multiple Publications organized by Territory.**

The Publication is an atomic unit within the Journal Release, not the Journal Release itself.

**Evidence:** GLOSSARY.md (Publication Package), User context (multiple publication types).

**Confidence:** HIGH.

## 8.3 Finding OBJ-003

**A Journal Release has composite identity: Reporting Period + Territory Set.**

Within one Reporting Period, there is exactly one Journal Release.

**Evidence:** GLOSSARY.md (Reporting Period), TERRITORIAL_MODEL.md (Territory structure).

**Confidence:** HIGH.

## 8.4 Finding OBJ-004

**Some parts of a Journal Release are mandatory, some optional, some temporary.**

Mandatory: Reporting Period, Territory Set, Emergency/Planned Outages (when exist).

Optional: Tomorrow publications, Graph publications, Technical publications.

Temporary: Tomorrow publications (disappear before end of day).

**Evidence:** User context.

**Confidence:** MEDIUM — requires further validation.
