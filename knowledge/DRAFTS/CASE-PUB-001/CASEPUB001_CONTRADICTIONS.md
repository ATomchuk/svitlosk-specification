# 2. Contradiction Register

## 2.1 CONTRADICTION-001: Telegram-Centric vs CHARTER

### Statements in Conflict

**Statement A (TELEGRAM_PUBLISHING_CANONICAL_MODEL.md):**
> "The Telegram Publishing System transforms normalized outage data into deterministic Telegram Journals."

**Statement B (CHARTER.md):**
> "SvitloSk is not defined by a particular technology, platform or communication channel. It is defined by its mission."

### Analysis

Statement A assumes Telegram as the primary publication target.

Statement B explicitly rejects technology-specific definitions.

The Telegram-centric model was built BEFORE the CHARTER principle was fully recognized.

### Severity: HIGH

This contradiction affects the fundamental architecture.

### Status: UNRESOLVED

Requires architectural decision.

---

## 2.2 CONTRADICTION-002: Publication vs Journal Release as Primary

### Statements in Conflict

**Statement A (GLOSSARY.md):**
> "Publication: A public information message generated by the SvitloSk system for distribution through one or more supported communication channels."

**Statement B (User context):**
> "The real subject is publication of the SvitloSk Public Information Journal."

### Analysis

Statement A defines Publication as the atomic unit.

Statement B identifies the Journal (and its Releases) as the primary product.

The Glossary defines Publication but does NOT define Journal Release.

### Severity: MEDIUM

This is a terminology gap, not a logical contradiction.

### Status: UNRESOLVED

Requires terminology clarification.

---

## 2.3 CONTRADICTION-003: Issue vs Journal Release

### Statements in Conflict

**Statement A (TELEGRAM_PUBLICATION_LIFECYCLE.md):**
> "An Issue opens when the first Publication for a calendar day is generated. The Issue represents one editorial edition of the Journal for that day."

**Statement B (User context):**
> "One Journal Release may contain: Emergency outage publication, Planned outage publication, Graph of planned power schedule..."

### Analysis

Statement A defines "Issue" as one editorial edition for one day.

Statement B uses "Journal Release" for the same concept.

"Issue" is defined only in Telegram context.

"Journal Release" is channel-independent.

### Severity: MEDIUM

Terminology inconsistency across contexts.

### Status: UNRESOLVED

Requires terminology resolution.

---

## 2.4 CONTRADICTION-004: Publication Package vs Journal Release

### Statements in Conflict

**Statement A (GLOSSARY.md):**
> "Publication Package: The complete collection of Publications generated during one Reporting Period."

**Statement B (User context):**
> "One Journal Release may contain: Emergency outage publication, Planned outage publication..."

### Analysis

Statement A defines "Publication Package" as the collection.

Statement B uses "Journal Release" for the same concept.

"Publication Package" is a technical term.

"Journal Release" is a domain term.

### Severity: LOW

Both terms describe the same concept at different abstraction levels.

### Status: UNRESOLVED

Requires terminology alignment.

---

## 2.5 CONTRADICTION-005: Lifecycle Scope

### Statements in Conflict

**Statement A (GLOSSARY.md):**
> "Publication Lifecycle: The complete lifecycle of a Publication."

**Statement B (TELEGRAM_PUBLICATION_LIFECYCLE.md):**
> "An Issue opens when the first Publication for a calendar day is generated."

### Analysis

Statement A defines lifecycle for individual Publications.

Statement B defines lifecycle for Issues (daily editions).

There is NO explicit lifecycle for Journal Releases.

### Severity: HIGH

Lifecycle gap for the primary object.

### Status: UNRESOLVED

Requires lifecycle definition.

---

## 2.6 CONTRADICTION-006: Publisher as Role vs Component

### Statements in Conflict

**Statement A (GLOSSARY.md):**
> "Publisher: The logical Role responsible for preparing and distributing Publications."

**Statement B (TELEGRAM_PUBLISHING_CANONICAL_MODEL.md):**
> "Publication Engine — Transforms normalized dataset into deterministic Publication Package. Implements the Publisher Role."

### Analysis

Statement A defines Publisher as a ROLE.

Statement B defines Publication Engine as a COMPONENT implementing the Role.

This is a valid Component vs Role distinction (per ADR-001).

### Severity: LOW

This is NOT a contradiction — it's a valid architectural pattern.

### Status: RESOLVED

No action required.

---

## 2.7 CONTRADICTION-007: Channel Independence vs Channel Specificity

### Statements in Conflict

**Statement A (CHARTER.md):**
> "The implementation technology of the Journal may evolve without changing its role within the project."

**Statement B (TELEGRAM_PUBLISHING_CANONICAL_MODEL.md):**
> "Supported publication channels include: Telegram Public Information Journal; Progressive Web Application (PWA)."

### Analysis

Statement A says technology may evolve.

Statement B lists specific channels.

This is not a contradiction — Statement B is an IMPLEMENTATION of Statement A.

### Severity: LOW

This is NOT a contradiction — it's a valid implementation choice.

### Status: RESOLVED

No action required.

---

## 2.8 CONTRADICTION-008: Mandatory vs Optional Parts

### Statements in Conflict

**Statement A (User context):**
> "One Journal Release may contain: Emergency outage publication (today), Planned outage publication (today)..."

**Statement B (User context):**
> "Graph of planned power schedule for tomorrow (12 subqueues, only when such graph exists)"

### Analysis

Statement A lists parts that MAY exist.

Statement B specifies a part that exists ONLY WHEN a condition is met.

This is not a contradiction — it's a specification of conditional content.

### Severity: LOW

This is NOT a contradiction — it's a valid conditional specification.

### Status: RESOLVED

No action required.

# 4. Resolution Requirements

## 4.1 High Severity Resolutions

### CONTRADICTION-001 Resolution Requirements

- Determine if Telegram-centric model should be generalized
- Determine if Publishing Kernel concept is needed
- Determine how to support multiple channels

### CONTRADICTION-005 Resolution Requirements

- Define Journal Release lifecycle explicitly
- Determine relationship between Journal Release lifecycle and Publication lifecycle
- Determine how lifecycle interacts with channels

## 4.2 Medium Severity Resolutions

### CONTRADICTION-002 Resolution Requirements

- Clarify if Publication or Journal Release is the primary object
- Update Glossary if Journal Release becomes canonical

### CONTRADICTION-003 Resolution Requirements

- Determine if "Issue" should be generalized to "Journal Release"
- Determine if both terms should coexist

# 5. Findings

**Eight contradictions were identified during the investigation.**

CHARTER.md explicitly rejects technology-specific definitions, but the current architecture assumes Telegram as primary.

**Evidence:** CHARTER.md §1, TELEGRAM_PUBLISHING_CANONICAL_MODEL.md §3.

**Confidence:** HIGH.

There is no explicit lifecycle for Journal Releases, which is the primary object.

**Evidence:** Analysis of GLOSSARY.md, TELEGRAM_PUBLICATION_LIFECYCLE.md.

**Confidence:** HIGH.
