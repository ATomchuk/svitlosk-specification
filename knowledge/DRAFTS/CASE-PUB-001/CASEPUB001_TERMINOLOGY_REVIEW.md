# 1. Purpose

This document reviews current project concepts in light of the architectural reconstruction, identifying which concepts become questionable, which may disappear, and which new concepts appear naturally.

Following Research Method V1.0, this analysis is presented without recommendations.

---

# 2. Concept Review

## 2.1 Publication

### Current Definition (GLOSSARY.md)

> "A public information message generated by the SvitloSk system for distribution through one or more supported communication channels."

### Review

**Status:** VALID

Publication remains a valid concept as the atomic unit of information.

However, Publication is NOT the primary object — it is a PART of the Journal Release.

**Impact:** No change required. Publication remains valid as an atomic unit.

---

## 2.2 Issue

### Current Definition (TELEGRAM_PUBLICATION_LIFECYCLE.md)

> "An Issue opens when the first Publication for a calendar day is generated. The Issue represents one editorial edition of the Journal for that day."

### Review

**Status:** QUESTIONABLE

"Issue" is defined only in Telegram context.

The concept is VALID (one editorial edition for one day), but the term is Telegram-specific.

**Impact:** "Issue" may need to be generalized to "Journal Release" or kept as a channel-specific term.

---

## 2.3 Journal

### Current Definition (CHARTER.md)

> "The SvitloSk Public Information Journal is the primary public communication channel of the project."

### Review

**Status:** VALID

Journal remains valid as the ongoing publication stream.

The Journal is the ABSTRACT concept; Journal Releases are the CONCRETE instantiations.

**Impact:** No change required. Journal remains valid.

---

## 2.4 Publisher

### Current Definition (GLOSSARY.md)

> "Publisher: The logical Role responsible for preparing and distributing Publications."

### Review

**Status:** VALID but QUESTIONABLE

Publisher is defined as a Role, which is valid.

However, the current implementation (Publication Engine) is Telegram-specific.

**Impact:** Publisher Role remains valid. Implementation may need generalization.

---

## 2.5 Publication Engine

### Current Definition (GLOSSARY.md)

> "Publication Engine: A software Component implementing the Publisher Role."

### Review

**Status:** QUESTIONABLE

Publication Engine is currently Telegram-specific.

If a Publishing Kernel is introduced, Publication Engine may become a channel-specific adapter.

**Impact:** Publication Engine may be generalized to Publishing Kernel, with channel-specific adapters.

---

## 2.6 Telegram Publisher

### Current Definition (TELEGRAM_PUBLISHING_CANONICAL_MODEL.md)

> "Telegram Publisher — Telegram-specific implementation of Publisher Role. Handles Telegram Bot API interaction and message delivery."

### Review

**Status:** VALID

Telegram Publisher is a valid channel-specific component.

If multi-channel architecture is adopted, Telegram Publisher becomes one of several channel adapters.

**Impact:** No change required. Telegram Publisher remains valid as a channel adapter.

---

## 2.7 Telegram Message

### Current Usage

Telegram Message is the unit of delivery to the Telegram channel.

### Review

**Status:** VALID but CHANNEL-SPECIFIC

Telegram Message is a Telegram-specific concept.

Other channels will have their own message concepts (Facebook Post, PWA notification).

**Impact:** Telegram Message remains valid within Telegram context.

---

## 2.8 Facebook Post

### Current Usage

Not yet defined in the project.

### Review

**Status:** FUTURE CONCEPT

Facebook Post will be the unit of delivery to the Facebook channel.

It is analogous to Telegram Message but for Facebook.

**Impact:** Will need definition when Facebook channel is implemented.

---

## 2.9 Release

### Current Usage

Not defined in any canonical document.

Used in user context as "Journal Release."

### Review

**Status:** NEW CONCEPT

"Release" is used in user context to describe one daily edition of the Journal.

It is synonymous with "Issue" but channel-independent.

**Impact:** May need formal definition if adopted as canonical terminology.

---

## 2.10 Archive

### Current Definition (GLOSSARY.md)

> "Historical Archive: The persistent storage of historical Interpretation Results and Publications."

### Review

**Status:** VALID

Archive remains valid as the historical storage concept.

**Impact:** No change required.

---

## 2.11 Representation

### Current Usage

Not defined in any canonical document.

Used in this investigation to describe channel-specific renderings of the Journal Release.

### Review

**Status:** NEW CONCEPT

"Representation" describes how a Journal Release is rendered for a specific channel.

Each channel has its own Representation of the same Journal Release.

**Impact:** May need formal definition if adopted as canonical terminology.

# 6. Findings

## 6.1 Finding TERM-001

**Five concepts remain valid without change.**

Publication, Journal, Archive, Telegram Publisher, Telegram Message remain valid.

**Evidence:** GLOSSARY.md, CHARTER.md, TELEGRAM_PUBLISHING_CANONICAL_MODEL.md.

**Confidence:** HIGH.

## 6.2 Finding TERM-002

**Three concepts become questionable.**

Issue, Publisher, Publication Engine may need redefinition or generalization.

**Evidence:** Analysis of GLOSSARY.md, TELEGRAM_PUBLICATION_LIFECYCLE.md, TELEGRAM_PUBLISHING_CANONICAL_MODEL.md.

**Confidence:** HIGH.

## 6.3 Finding TERM-003

**One concept may disappear.**

Publication Package may be superseded by Journal Release.

**Evidence:** GLOSSARY.md (Publication Package), User context (Journal Release).

**Confidence:** MEDIUM.

## 6.4 Finding TERM-004

**Four new concepts appear naturally.**

Journal Release, Publishing Kernel, Channel Adapter, Representation appear from the architectural reconstruction.

**Evidence:** CASEPUB001_OBJECT_RECONSTRUCTION.md, CASEPUB001_KERNEL_ANALYSIS.md.

**Confidence:** HIGH.
