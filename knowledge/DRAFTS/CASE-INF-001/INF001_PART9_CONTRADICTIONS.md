---

# 2. Contradictions vs Refinements vs Evolutions

## 2.1 Contradiction

**Definition:** Two statements that cannot both be true.

**Example:** "X is true" AND "X is false"

## 2.2 Refinement

**Definition:** One statement provides more detail than another.

**Example:** "X exists" → "X has properties A, B, C"

## 2.3 Evolution

**Definition:** One statement updates or extends another.

**Example:** "X was true in the past" → "X is now different"

# 3. Contradictions

## 3.1 CONTRADICTION-001: Single Parser vs Two Parsers

### Repository Statement

> "Parser: A software Component responsible for retrieving Open Data from approved Data Sources." (GLOSSARY §2)

**Implication:** Single Parser component.

### Architect Intent Statement

> "Parser A: Collect official Queue Schedule."

> "Parser B: Collect planned and emergency outages."

> "These parsers are independent."

**Implication:** Two independent Parser components.

### Classification: CONTRADICTION

Repository describes ONE Parser.

Architect Intent describes TWO independent Parsers.

These cannot both be true simultaneously.

### Severity: HIGH

---

## 3.2 CONTRADICTION-002: Single Information Flow vs Three Flows

### Repository Statement

**Implication:** Single information pipeline.

### Architect Intent Statement

> "The system contains THREE independent information domains."

> "Queue Schedule → PWA."

> "Planned/Emergency Outages → Journal."

**Implication:** Three independent information flows.

### Classification: CONTRADICTION

Repository describes ONE pipeline.

Architect Intent describes THREE independent flows.

These cannot both be true simultaneously.

### Severity: HIGH

## 3.3 CONTRADICTION-003: Publication as Universal vs Domain-Specific Distribution

### Repository Statement

> "Publication: A public information message generated by the SvitloSk system for distribution through one or more supported communication channels." (GLOSSARY §3)

**Implication:** All information is distributed as Publications.

### Architect Intent Statement

> "The PWA visualizes primarily Domain A."

> "Push notifications belong ONLY to Queue Schedule."

**Implication:** Queue Schedule is distributed via PWA Timeline and Push Notifications, not Publications.

### Classification: CONTRADICTION

Repository implies all information is distributed as Publications.

Architect Intent implies Queue Schedule uses different distribution mechanisms.

These cannot both be true simultaneously.

### Severity: MEDIUM

---

## 3.4 CONTRADICTION-004: Channel Ownership vs Channel Independence

### Repository Statement

> "Telegram Journal: The official public communication channel operated by the SvitloSk Project." (GLOSSARY §3)

**Implication:** Telegram Journal is a channel that operates Publications.

### Architect Intent Statement

> "Telegram and Facebook DO NOT own information."

> "They receive prepared representations."

> "Publishing is channel-independent."

**Implication:** Channels do not own information — they receive representations.

### Classification: CONTRADICTION

Repository implies Telegram Journal "operates" Publications.

Architect Intent implies channels do not own information.

These cannot both be true simultaneously.

### Severity: MEDIUM

## 3.5 CONTRADICTION-005: Journal as Primary vs Multiple Products

### Repository Statement

> "The SvitloSk Public Information Journal is the primary public communication channel of the project." (CHARTER §10)

**Implication:** Journal is the PRIMARY channel.

### Architect Intent Statement

> "The PWA visualizes primarily Domain A."

> "The Public Information Journal contains information from Domains A, B and C."

**Implication:** PWA and Journal are BOTH products, neither is primary.

### Classification: CONTRADICTION

Repository says Journal is PRIMARY.

Architect Intent implies PWA is equally important.

These cannot both be true simultaneously.

### Severity: MEDIUM

---

## 3.6 CONTRADICTION-006: Address as Universal vs Queue-Based Identity

### Repository Statement

> "Address: A geographical location published by the official Data Source." (GLOSSARY §4)

> "Territory: A publication unit representing either: the Administrative Centre; one Starosta District." (GLOSSARY §4)

**Implication:** Address and Territory are the primary identity dimensions.

### Architect Intent Statement

> "Queue Schedule: 6 queues, 12 subqueues, hourly, not address based."

**Implication:** Queue Schedule uses Queue/Subqueue identity, NOT Address identity.

### Classification: REFINEMENT

Repository defines Address and Territory.

Architect Intent adds Queue/Subqueue as additional identity dimensions.

This is a REFINEMENT, not a contradiction.

### Severity: LOW

---

## 3.7 CONTRADICTION-007: Daily as Universal vs Hourly

### Repository Statement

> "Reporting Period: The time interval for which Publications are generated." (GLOSSARY §2)

> "A Reporting Period MAY correspond to: a calendar day..." (GLOSSARY §2)

**Implication:** Reporting Period is typically daily.

### Architect Intent Statement

> "Queue Schedule: hourly."

**Implication:** Queue Schedule changes hourly, not daily.

### Classification: REFINEMENT

Repository allows Reporting Period to be flexible.

Architect Intent specifies hourly for Queue Schedule.

This is a REFINEMENT, not a contradiction.

### Severity: LOW

---

## 3.8 CONTRADICTION-008: Release as Undefined

### Repository Statement

No definition of "Release" in canonical documents.

### Previous Investigation Statement

> "Release is a COMPOSITE ENTITY." (CASE-PUB-001)

**Implication:** Release is a concept.

### Architect Intent Statement

No mention of "Release."

### Classification: EVOLUTION

Release appeared in previous investigations but is not in repository or Architect Intent.

This may be an EVOLUTION of terminology.

### Severity: LOW
